# 스타트업 론칭 (startup-launcher)

> MoAI-Cowork v1.0.0 하네스 참고자료

## 개요

시장 분석, 비즈니스 모델링, MVP 아키텍처, 피치덱 작성, 론칭 검수를 통해 스타트업 론칭을 협력하여 계획하는 하네스입니다.

## 전문가 역할

- **비즈니스 모델러**: Business Model Canvas 작성, 수익 모델 설계, 단위경제 분석, 재무 예측 구축
  - Business Model Canvas: 9개 블록을 구체적으로 채워 전체 비즈니스 구조 시각화
  - 수익 모델 설계: 가격 책정 방식(구독/거래 수수료/광고/프리미엄 등), 가격 전략, 수익 다각화
  - 단위경제: CAC, LTV, LTV/CAC 비율, 회수 기간, 매출총이익 계산
  - 재무 예측: 3년 수익/비용/손익 계산 (낙관적/기본/보수적 시나리오)
  - 자금 요구: 연소비율, 활주로, 단계별 투자 필요액

- **론칭 검수자**: QA 역할, 시장 분석-비즈니스 모델-MVP-피치덱 간 논리 일관성 교차 검증, 투자 준비도와 실행 가능성 평가
  - 시장 <> 비즈니스 모델 일관성: 시장 규모가 수익 예측을 뒷받침하는가?
  - 비즈니스 모델 <> MVP 일관성: MVP가 핵심 수익 모델을 검증할 수 있는가?
  - 전체 <> 피치덱 일관성: 피치덱 수치가 분석 리포트와 일치하는가?
  - 투자 준비도 평가: 투자자 미팅 수준에 적합한가? (VC 어소시에이트 관점)
  - 실행 가능성: 제안된 로드맵이 현실적인가? 리스크를 파악했는가?

- **시장 분석가**: 스타트업 시장 분석가, 아이디어 검증, 시장 규모 산정(TAM/SAM/SOM), 경쟁사 분석, 고객 페르소나 정의, PMF 가설 수립
  - 아이디어 검증: 문제-솔루션 핏 분석, 기존 대안과의 차별화
  - 시장 규모 산정: 톱다운(TAM > SAM > SOM) + 바텀업 교차 검증
  - 경쟁사 분석: 직접/간접 경쟁사 맵핑, 포지셔닝 맵, 진입장벽 분석
  - 고객 페르소나: 초기 타겟 고객의 인구통계, 통증점, 행동 정의
  - PMF 가설: 검증할 핵심 가설과 검증 방법(인터뷰, 설문, 랜딩 페이지 등) 제안

- **MVP 아키텍트**: 핵심 기능 우선순위 지정, 기술 스택 선택, 개발 로드맵 작성, 사용자 흐름 설계
  - 핵심 기능 우선순위: MoSCoW 방식 또는 RICE 스코어링으로 기능 우선순위 지정
  - 기술 스택 선택: 개발 속도, 비용, 확장성을 고려하여 최적 스택 선택
  - 사용자 흐름 설계: 핵심 사용자 여정 맵핑, 주요 화면 와이어프레임
  - 개발 로드맵: 단계별 개발 계획, 마일스톤, 일정, 리소스 요구
  - 성공 지표 정의: MVP를 통해 PMF 가설을 검증할 정량 지표 정의

- **피치 작성자**: 투자자 프레젠테이션 스토리라인, 슬라이드 구조, 핵심 내러티브 설계
  - 스토리라인 설계: 문제 > 솔루션 > 시장 > 트랙션 > 비즈니스 모델 > 팀 > 펀딩 내러티브 호
  - 슬라이드 구조: 투자자 기대치 반영한 10-15장 피치덱
  - 데이터 시각화: 각 슬라이드별 복잡한 데이터를 명확하고 임팩트 있는 시각으로 변환
  - 투자자 Q&A 준비: 주요 질문 예상 및 명확한 답변 준비
  - 다양한 형식: 전체 덱, 익스큐티브 요약(2쪽), 엘리베이터 피치(30초)

## 워크플로우

### Phase 1: 준비 (오케스트레이터 직접 수행)

1. 사용자 입력에서 추출:
   - **스타트업 아이디어**: 해결할 문제, 제안된 솔루션
   - **타겟 시장**: 산업, 지역, 고객 세분
   - **단계**: 아이디어/검증/MVP/성장
   - **자금 목표** (선택): 투자 라운드, 목표 액수
   - **기존 자산** (선택): 시장 조사, 비즈니스 계획, 프로토타입 등

2. `_workspace/` 디렉토리 생성
3. 입력을 정리하여 `_workspace/00_input.md`로 저장
4. 기존 파일 제공 시 `_workspace/`로 복사 후 해당 Phase 건너뛰기
5. 요청 범위에 기반해 **실행 모드** 결정

### Phase 2: 팀 편성 및 실행

| 순서 | 작업 | 담당자 | 의존사항 | 산출물 |
|------|------|--------|---------|--------|
| 1 | 시장 검증 | 시장 분석가 | 없음 | `_workspace/01_market_validation.md` |
| 2 | 비즈니스 모델 | 비즈니스 모델러 | 작업 1 | `_workspace/02_business_model.md` |
| 3 | MVP 설계 | MVP 아키텍트 | 작업 2 | `_workspace/03_mvp_design.md` |
| 4 | 피치덱 | 피치 작성자 | 작업 1, 2, 3 | `_workspace/04_pitch_deck.md` |
| 5 | 론칭 검수 | 론칭 검수자 | 작업 4 | `_workspace/05_review_report.md` |

**에이전트 간 커뮤니케이션:**
- 시장 분석가 완료 → 시장 데이터 비즈니스 모델러에게, 통증점 MVP 아키텍트에게 전달
- 비즈니스 모델러 완료 → 재무 정보 MVP 아키텍트와 피치 작성자에게 전달
- MVP 아키텍트 완료 → MVP 계획 피치 작성자에게 전달
- 피치 작성자 완료 → 덱 론칭 검수자에게 전달
- 검수자 모든 산출물 교차 검증, 🔴 반드시 수정할 항목 발견 시 담당자에게 수정 요청 → 재작업 → 재검증 (최대 2라운드)

### Phase 3: 통합 및 최종 산출물

1. `_workspace/` 내 모든 파일 검증
2. 모든 🔴 반드시 수정할 항목 해결 확인
3. 사용자에게 최종 요약 보고

## 산출물

`_workspace/` 디렉토리에 저장된 모든 결과물:
- `00_input.md` — 사용자 입력과 스타트업 아이디어
- `01_market_validation.md` — 시장 분석 리포트
- `02_business_model.md` — 비즈니스 모델 및 수익 계획
- `03_mvp_design.md` — MVP 아키텍처 및 개발 로드맵
- `04_pitch_deck.md` — 피치덱 콘텐츠
- `05_review_report.md` — 론칭 실행 가능성 검수 리포트
