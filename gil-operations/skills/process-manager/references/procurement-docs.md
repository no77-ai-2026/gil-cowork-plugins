조달 문서 세트(요건 정의→공급업체 비교→평가 기준→계약 검토→수락 기준)를 에이전트 팀이 협업하여 작성하는 하네스입니다.

## 나라장터(G2B) 공공조달 가이드

### 나라장터란

나라장터(G2B, Government to Business)는 조달청이 운영하는 국가 전자조달 시스템으로, 모든 공공기관의 물품·용역·공사 구매를 처리합니다. 2026년 기준 연간 약 200조 원 규모의 조달이 이 시스템을 통해 이루어집니다.

---

### 나라장터 등록 절차

#### 1단계: 업체 등록

1. 나라장터(g2b.go.kr) 접속 → 공급자 회원 가입
2. 사업자등록증 사본 업로드
3. 대표자 공인인증서(범용) 또는 조달청 G-PIN 준비
4. 입찰 참가 자격 등록 신청 (조달청 지사 방문 또는 온라인)
5. 등록 완료 후 조달업체 코드 발급 (처리 기간: 3-5 영업일)

#### 2단계: 입찰 참가

1. 나라장터 공고 검색 (조달 유형·분류 번호·발주기관 필터)
2. 입찰 공고 상세 확인 (납품 조건, 규격, 평가 기준)
3. 입찰 참가 신청 및 보증금 납부 (입찰보증금: 입찰 금액의 5%)
4. 전자입찰서 제출 (마감 시각 30분 전까지 권장)
5. 개찰 결과 확인

#### 3단계: 계약 체결

1. 낙찰 통보 수신 후 5일 이내 계약 서류 제출
2. 계약보증금 납부 (계약금액의 10%)
3. 전자계약 체결 (나라장터 시스템 내 서명)
4. 납품 → 검사·검수 → 대금 청구 순서

### 조달 유형

| 유형 | 설명 | 적용 기준 |
|------|------|-----------|
| 일반경쟁 | 자격 요건을 갖춘 누구나 참가 | 제한 없음 (원칙) |
| 제한경쟁 | 지역·실적·규모 등 참가 자격 제한 | 발주기관이 요건 설정 |
| 수의계약 | 경쟁 없이 특정 업체와 계약 | 2,000만 원 미만 또는 긴급 사유 |
| 지명경쟁 | 발주기관이 지명한 업체만 참가 | 특수 기술·보안 요구 시 |
| 다수공급자계약(MAS) | 조달청 사전 계약, 수요기관 직접 구매 | 소프트웨어·IT 장비 등 |

### 필수 서류 목록

#### 입찰 참가 자격 등록 시

- [ ] 사업자등록증 사본
- [ ] 법인등기부등본 (법인의 경우)
- [ ] 납세증명서 (국세·지방세 완납 확인)
- [ ] 건강보험·국민연금 완납 증명서
- [ ] 업종별 면허·등록증 (해당 업종)

#### 입찰 참가 시

- [ ] 입찰 참가 신청서 (전자 제출)
- [ ] 사업자등록증 사본
- [ ] 수행 실적 증명서 (제한 경쟁 시)
- [ ] 입찰 보증보험증권 (현금 대체 가능)
- [ ] 기술·가격 제안서 (협상에 의한 계약 시)

---

### 입찰 참가 자격 요건

- 사업자등록 상태 (폐업·정지 불가)
- 국세·지방세 완납 확인 (입찰일 기준 최근 3개월)
- 부정당업자 제재 기간 아닐 것
- 업종별 면허·등록 유효 (건설업: 건설업 등록증, IT: 소프트웨어사업자 신고 등)
- 중소기업 우대: 중소기업 확인서로 가점 또는 분리 공고 혜택

---

### 전자 입찰 시스템 사용법 요약

1. **공인인증서 준비**: 범용 공인인증서 또는 G-PIN 필수
2. **나라장터 보안 모듈 설치**: 처음 접속 시 ActiveX 또는 보안 플러그인 설치
3. **입찰 공고 검색**: 메인 → 입찰정보 → 조건별 검색 (품명/공고번호/발주기관)
4. **전자입찰 참가**: 공고 상세 → 입찰 참가 신청 → 금액 입력 → 전자서명 제출
5. **결과 확인**: 개찰 후 입찰 내역 → 결과 조회

> 참고: 2026년부터 일부 공공기관은 모바일 앱(나라장터 앱)을 통한 입찰 결과 확인 지원

## 전문가 역할

- **요건 정의 전문가**: 구매 필요 사항을 명확하고 검증 가능한 요구사항으로 정의
  - 비즈니스 요구사항: 해결할 비즈니스 문제 명확히 정의
  - 기술 요구사항: 필요한 기능, 성능, 호환성 구체화
  - 필수 vs 선택: Must-have와 Nice-to-have 요구사항 분명히 구분
  - 우선순위: 각 요구사항의 중요도 가중치 정의
  - 검증 기준: 각 요구사항이 충족되었는지 검증하는 방법 정의

- **공급업체 평가 전문가**: 공급업체 자격 기준, 평가 프레임워크, 리스크 평가 설계
  - 평가 기준: 가격, 품질, 기술력, 신뢰성, 지원 능력 등 평가 항목 정의
  - 가중치 설정: 각 기준의 상대적 중요도 설정
  - 리스크 평가: 재무 안정성, 공급 안정성, 기술 위험 평가
  - 참고 자료: 제품/서비스 동향, 경쟁사 비교, 산업 표준 조사
  - 점수카드: 평가 결과를 객관적으로 비교하는 점수카드 설계

- **계약 검토 전문가**: 계약서 조항, 리스크 조항, SLA, 지적재산권, 종료 조건 분석
  - 표준 계약 구조: 구매 유형별 적절한 표준 계약 구조 제시
  - 리스크 조항: 페널티, 손해배상, 불가항력, 종료 조항 분석
  - SLA/SLO 정의: 서비스 수준 합의서 지표, 목표, 페널티 설계
  - 지적재산권: 납품물 IP, 라이선스, 데이터 소유권 관련 조항 검토
  - 협상 전략: 구매자 관점에서 유리한 조건 확보 전략 수립

- **수락 기준 개발자**: 납품물의 객관적 수락을 위한 검사 항목, 시험 절차, 합격 기준 설계
  - 검사 항목: 요구사항 명세서의 각 항목을 검사 가능한 항목으로 변환
  - 시험 절차: 각 검사 항목별 구체적 시험 방법 및 절차 설계
  - 합격 기준: 합격/조건부/부합격 판정 기준 정량화
  - 수락 일정: 문서 검토→기능 시험→성능 시험→최종 승인 단계별 일정
  - 결함 처리: 부합격 시 시정 요청, 재시험, 반품 절차 정의

## 워크플로우

### Phase 1: 준비 (오케스트레이터 직접 수행)

1. 사용자 입력에서 추출:
   - **구매 항목**: 상품/서비스, 규모, 예산
   - **현황**: 기존 공급업체, 현재 문제점
   - **요구사항**: 주요 필요 기능, 제약사항
   - **기존 자료** (선택): 기존 계약, 요구사항 문서

2. `_workspace/` 디렉토리 생성
3. 입력값 정리하여 `_workspace/00_input.md`에 저장
4. 요청 범위에 따라 **실행 모드 결정**

### Phase 2: 팀 구성 및 실행

| 순서 | 작업 | 담당 | 의존성 | 산출물 |
|------|------|------|--------|--------|
| 1 | 요건 정의 | 요건 전문가 | 없음 | `_workspace/01_requirements_spec.md` |
| 2 | 공급업체 평가 | 평가 전문가 | 작업 1 | `_workspace/02_vendor_evaluation.md` |
| 3 | 계약 검토 | 계약 전문가 | 작업 1 | `_workspace/03_contract_terms.md` |
| 4 | 수락 기준 | 수락 전문가 | 작업 1, 3 | `_workspace/04_acceptance_criteria.md` |

### Phase 3: 통합 및 최종 산출

1. 모든 문서 검증 및 일관성 확인
2. 문서 세트 최종 검토
3. 조달 프로세스 가이드 작성

## 산출물

모든 산출물은 `_workspace/`에 저장됩니다:
- `00_input.md` — 사용자 입력 정리
- `01_requirements_spec.md` — 요구사항 명세서
- `02_vendor_evaluation.md` — 공급업체 평가 기준 및 점수카드
- `03_contract_terms.md` — 표준 계약 조항 및 협상 전략
- `04_acceptance_criteria.md` — 수락 기준 및 시험 절차

## 오류 처리

| 오류 유형 | 대응 전략 |
|----------|----------|
| 요구사항 불명확 | 요건 전문가가 상품류별 표준 템플릿 제공 |
| 공급업체 정보 부족 | 평가 전문가가 3개 후보사 가정 평가 |
| 법률 조항 불명확 | 계약 전문가가 업계 표준 계약서 샘플 제공 |
| 수락 시험 설계 어려움 | 수락 전문가가 단계적 시험 계획 수립 |
| 문서 복잡도 높음 | 단순화된 버전(기본)과 상세 버전 제공 |
