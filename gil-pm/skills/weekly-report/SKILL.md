# Weekly Report — 한국 팀 주간보고(WBR) 자동 생성

> gil-pm | 한국 주간보고 문화 + 글로벌 베스트 프랙티스

## 개요

월요 오전 WBR(Weekly Business Review) 또는 금요 마무리 보고용 주간 리포트를 자동 작성합니다. 한국 직장 문화에서 정착된 6섹션 표준(KPI / Done / In Progress / Plan / Risks / Decisions)을 기본으로 하며, 임원용 1pager와 팀 내부 상세 두 버전을 동시 생성합니다.

## 트리거 키워드

주간보고 WBR 위클리 리포트 주간 회의자료 주간 업무보고 weekly report 임원 주간보고 팀 주간보고

## 실전 주간보고·회의·회고 패턴 (한국 팀)

주간보고는 단독 산출물이 아니라 **WBR 미팅 + 회고 + 다음 주 우선순위 결정**의 종합 보조 자료입니다. 한국 조직 문화에 최적화된 노하우입니다.

### 6개 핵심 패턴

**① 한국 WBR 표준 6섹션 (이미 적용 + vault 검증)**

| # | 섹션 | 한국 실무 함정 |
|---|---|---|
| 1 | 이번 주 핵심 (KPI) | 목표 대비 % 표시 필수 — 절대값만 보고 ❌ |
| 2 | 완료 (Done) | "노력함" X, **임팩트 정량** (-30% 비용, +15% 유저) |
| 3 | 진행 중 (In Progress) | 진행률 % + **다음 주 완료 예정 여부** 명시 |
| 4 | 차주 계획 (Plan) | P0/P1/P2 우선순위 — **시간 단위 추정 금지** |
| 5 | 이슈·리스크 | 단순 나열 X — **영향도 + 대응안** 동반 |
| 6 | 의사결정 요청 | **옵션 2-3개 제시** — 단순 질문 X |

**② 회고 프레임워크 KPT·4Ls (한국 기획자 검증)**

주간보고에 회고 섹션 통합 권장. 2종 프레임워크 중 선택:

**KPT (가장 흔함)**:
- **K**eep: 지속할 것
- **P**roblem: 개선 필요한 것
- **T**ry: 새로 시도할 것

**4Ls (감정 포함)**:
- **L**ikes: 좋았던 점·성과
- **L**earned: 배운 점 (성공·실패에서)
- **L**acked: 부족했던 점
- **L**onged for: 바랐던 점

> 회고는 머릿속 추상을 **'읽을 수 있는 언어'** 로 표현 → 기획자의 필수 역량 "소통 능력" 훈련.

**③ 의사결정 요청 표준 포맷 (한국 임원 보고)**

```markdown
## 의사결정 요청 — [주제]

### 배경
[1~2줄 맥락]

### 옵션
| 옵션 | 장점 | 단점 | 비용·일정 |
|---|---|---|---|
| A (권장) | ... | ... | ... |
| B | ... | ... | ... |
| C | ... | ... | ... |

### 권장 사유
[1~2줄 핵심]

### 결정 마감일
[명시]
```

**"고민됩니다, 어떻게 할까요"식 질문 X** — 임원은 옵션 + 권장을 함께 보길 원함.

**④ 스크럼 주기 결정 매트릭스 (주간보고 입력)**

| 프로젝트 단계 | 권장 스크럼 주기 | 주간보고 입력 |
|---|---|---|
| 빠른 실행·MVP | 데일리 스크럼 | 매일 노트 → 주간 요약 |
| 일반 개발·기획 | 주 2-3회 | 스크럼 메모 직접 통합 |
| 검토·승인 중심 | 주 1회 | 주 1회 + 비동기 업데이트 |
| 장기 R&D | 격주 | 격주 + 마일스톤 보고 |

스크럼 = 주간보고의 raw input. 데일리 스크럼 메모를 미리 정리해두면 주간보고는 30분 안에 작성 가능.

**⑤ 톤 자동 매핑 (한국 조직 위계 반영)**

| 보고 대상 | 톤 | 어조 예시 |
|---|---|---|
| **C-Level (CEO/CFO/COO)** | 격식체 + 결론 우선 | "-합니다", "-로 판단됩니다" |
| **부서장·팀장** | 격식체 | "-합니다", 정중한 보고체 |
| **팀 내부 (Slack)** | 해요체 | "-했어요", "-할 예정이에요" |
| **외부 클라이언트** | 격식체 + 존경어 | "-하셨습니다", "-해 주시면 감사하겠습니다" |
| **회고 (팀 내)** | 자기 성찰체 | "-했더라고요", "-를 배웠어요" |

혼합 보고 시 두 버전 동시 출력 — 1pager (임원) + 풀 버전 (팀).

**⑥ 회의 진행법 3종 통합 (주간보고 산출물의 활용)**

| 회의 | 주간보고 활용 |
|---|---|
| **기획 리뷰** | 진행 중 섹션 + 의사결정 요청 항목 — 큰 골자 vs 실행안 분기 |
| **스크럼** | KPI·진행률 섹션 — 현재 이슈·블로커 빠른 공유 |
| **회고** | KPT·4Ls 섹션 — 이번 주 회고 + 다음 주 Try |

기획 리뷰 핵심 4단:
1. **컨셉안 vs 실행안 구분** — 누구에게 무엇을 리뷰할지 사전 결정
2. **문서 미리 첨부 + 피드백 받고 싶은 부분 명시**
3. **즉시 반박 회피** — "충분히 고려한 뒤 추가 검토하겠습니다" 식으로 감정 충돌 방지
4. **회의록 당일 공유** — 청자 기억 휘발 방지

### 한국 주간보고 안티 패턴

- 일주일 행동 일기처럼 시간순 나열 (임팩트 X)
- KPI 목표값 없이 절대값만 ("MAU 12,000" — 목표 대비 모름)
- 우선순위 없이 "다음 주 할 일 10개" 나열
- 의사결정 요청을 단순 질문으로 ("어떻게 하면 될까요?")
- 이슈·리스크 섹션 비워두기 (위장 평화 신호)
- 회고·인사이트 없이 데이터만

### 노션·Linear·Asana 입력 시 데이터 자동 동기화 팁

- **Notion**: 데이터베이스 view를 `Done this week` / `In progress` / `Planned` 3 필터로 분리
- **Linear**: Cycle 단위로 이슈 자동 그룹화 → MCP 가용 시 직접 fetch
- **Asana**: Section별 정렬 + Status 필드로 진행률 계산

가용 도구 없을 때는 자유 텍스트 메모 + LLM 정리로 fallback.

> 💡 한국 주간보고 1년 효과 누적: **(1) 본인 업무 시각화 (2) 임원 의사결정 속도 향상 (3) 회고 습관화로 소통 능력 향상**. 잘 쓴 주간보고 1년이면 발화·문서 작성 능력 모두 변화.

## 워크플로우

### 1단계: 데이터 소스 자동 감지 (3-tier fallback)

| Tier | 소스 | 동작 |
|---|---|---|
| 1 | Notion·Linear·Asana MCP 가용 | 데이터베이스/이슈 자동 fetch |
| 2 | CSV/JSON 업로드 | 외부 도구 export 파싱 |
| 3 | 자유 텍스트 메모 | 사용자가 정리한 내용 그대로 활용 |

MCP 가용 시 자동 우선 활용. 사용자가 명시적으로 텍스트 입력 시 그대로 처리.

### 2단계: 6섹션 본문 생성 (한국 WBR 표준)

| # | 섹션 | 내용 |
|---|---|---|
| 1 | 이번 주 핵심 (KPI) | 사전 정의된 지표 달성치·목표 대비 % |
| 2 | 완료 (Done) | PR·티켓·기능 단위, 임팩트 정량 |
| 3 | 진행 중 (In Progress) | 다음 주 완료 예정 항목·진행률 |
| 4 | 차주 계획 (Plan) | priority labels (P0/P1/P2) — 시간 단위 추정 금지 |
| 5 | 이슈·리스크 | 외부 의존성, 일정 압박, 기술 부채 |
| 6 | 의사결정 요청 | 임원/PM 결정 필요 항목 + 옵션 |

### 3단계: 톤 자동 적용

사용자가 톤을 지정하지 않으면 대상별 자동 매핑:

- **임원 보고**: 격식체 ("-합니다", "-로 판단됩니다")
- **팀 내부 (Slack 등)**: 해요체 ("-했어요", "-할 예정이에요")
- **혼합**: 두 버전 동시 출력

### 4단계: 임원 1pager 별도 생성 (선택)

≤ 250 단어로 압축:
- KPI 헤드라인 1줄
- 완료 핵심 3개 (각 1줄, 정량)
- 리스크 1개
- 의사결정 요청 1-2개

### 5단계: 후처리 가이드

- **executive-summary**: 1pager를 더 압축된 C-level 보고로 변환
- **ai-slop-reviewer**: 격식체 톤 검수 (CLAUDE.local.md §3-2)
- **pptx-designer**: 슬라이드 발표 자료로 출력

## 출력 형식

```markdown
# 주간보고 — [팀명] (2026년 W18, 5/1~5/7)

## 1. 이번 주 핵심 (KPI)
- MAU: 12,450 (목표 12,000 대비 +3.8%)
- 신규 가입: 320 (목표 300 대비 +6.7%)

## 2. 완료 (Done)
- [Feature] 결제 페이지 v2 — PR #234 머지, 결제 성공률 +2.1pp
- [Bug] 로그인 500 에러 — 영향 사용자 ~30명, 핫픽스 배포

## 3. 진행 중 (In Progress)
- 알림 시스템 리팩토링 — 70% 완료, 차주 머지 목표

## 4. 차주 계획 (Plan)
- [P0] 결제 모듈 보안 감사 대응
- [P1] 마케팅 캠페인 페이지 제작
- [P2] 백엔드 의존성 업데이트

## 5. 이슈·리스크
- 외부 SMS API 응답 지연 → backup 채널 검토 필요
- 디자인팀 리소스 부족으로 차주 일부 작업 후순위

## 6. 의사결정 요청
- 보안 감사 대응 인력 추가 (옵션 A: 외주, B: 내부 재배치)
```

임원 1pager (≤ 250 단어):
```markdown
# Executive Brief — W18

**KPI**: MAU 12.4K (+3.8% vs 목표), 가입 320 (+6.7%)
**완료**: 결제 v2 머지(전환 +2.1pp) / 로그인 500 핫픽스 / 알림 70%
**리스크**: SMS API 지연 — 백업 채널 검토 중
**결정 요청**: 보안 감사 대응 — 외주 vs 내부 재배치
```

## 사용 예시

**예시 1 — Notion MCP 자동 fetch**
```
사용자: "이번 주 주간보고 만들어줘. 우리 팀 Notion DB 'Weekly Tasks' 참고."
→ Notion MCP로 W18 완료 태스크·진행 중 자동 수집
→ 6섹션 격식체 보고 + 임원 1pager 두 버전 출력
```

**예시 2 — 자유 텍스트 fallback (해요체)**
```
사용자: "이번 주 한 일 정리해줘. 결제 v2 머지했고, 로그인 버그 잡았고, 알림 리팩토링 70% 됐어."
→ 자유 텍스트 파싱 후 해요체 팀 내부 보고로 변환
```

## 주의사항

- **시간 추정 금지**: "다음 주 화요일까지 완료" 같은 직접 날짜·시간 추정은 priority labels로 변환
- **개인 평가 금지**: 팀원 개인 비난·칭찬은 본문에 자동 포함되지 않음 (필요 시 별도 1:1)
- **확인되지 않은 수치 금지**: KPI 데이터 출처(BI 도구·내부 DB) 명시 또는 [추정] 태그
- **NDA 정보 주의**: 외부 공유용 보고서 작성 시 회사 비밀 정보 자동 검사 (`gil-legal/nda-triage` 권고)
