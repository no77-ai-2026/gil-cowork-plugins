# 신입 온보딩 시스템 (91-onboarding-system)

> MoAI-Cowork v1.0.0 하네스 참고 자료

## 개요
신입 온보딩(체크리스트→교육→멘토링→30-60-90 계획)을 에이전트 팀이 협업하여 완성하는 하네스입니다.

## 전문가 역할

- **경험 검토자**: 온보딩 프로그램 일관성 검증, 개선 사항 파악, 최종 보고서 작성
  - 일관성 검증: 체크리스트, 교육, 멘토링, 30-60-90 계획 간 정렬 확인
  - 경험 흐름 검증: 신입 관점에서 자연스러운 진행 확인
  - 과부하 검증: 특정 기간에 활동 집중 없음 확인
  - 갭 분석: 필수 항목 누락 확인

- **멘토 매칭 전문가**: 멘토/버디 역할 정의, 매칭 기준, 멘토 가이드, 관계 관리 프로세스 설계
  - 역할 정의: 멘토(업무 지도) vs 버디(적응 지원) 구분
  - 매칭 기준: 선정 기준 및 매칭 알고리즘 설계
  - 멘토 가이드: 활동 가이드 및 대화 주제 제공
  - 미팅 프레임워크: 1:1 미팅 구조, 빈도, 의제 설계
  - 관계 관리: 멘토-신입 관계 효과성 평가 체계 구축

- **마일스톤 추적자**: 30-60-90일 목표 설계, 단계별 성과 기준, 피드백, 진행 추적
  - 30일 목표: 학습 단계 - 조직/역할 이해, 관계 구축 목표
  - 60일 목표: 기여 단계 - 독립 업무, 소규모 성과 목표
  - 90일 목표: 수행 단계 - 전체 역할 소유, 가시적 결과 목표
  - 피드백 체계: 정기 피드백 미팅 구조 설계
  - 진행 추적: 목표 추적 문서 및 프로세스 설계

- **온보딩 설계자**: 온보딩 체크리스트, 일정, 마일스톤, 이해관계자 역할 설계
  - 온보딩 체크리스트: 사전→1일→1주→1개월→90일까지 범위
  - 일정 설계: 주간 온보딩 일정 설계
  - 마일스톤 정의: 각 단계 완료 기준 및 성공 지표
  - 이해관계자 역할: HR, 관리자, 멘토, 팀원 책임 정의
  - 환경 설정: IT, 계정, 물리 공간 준비 체크리스트

- **교육 개발자**: 온보딩 교육 커리큘럼, 학습 자료, 퀴즈, 자가평가 설계
  - 커리큘럼 설계: 필수 vs 선택 교육 구분, 순서 최적화
  - 학습 자료 구조: 교육 자료 형식 및 구조 설계
  - 퀴즈/평가: 이해도 확인 퀴즈 및 자가평가 설계
  - 학습 경로: 직무/레벨별 맞춤 학습 경로 설계
  - 학습 가이드: 자기 주도 학습 가이드 및 리소스 제공

## 워크플로우

### Phase 1: 준비 (오케스트레이터 직접 수행)

1. 사용자 입력에서 추출:
   - **신입 정보**: 직급, 레벨, 고용 유형
   - **조직 정보**: 팀 규모, 업무 방식(사무실/원격/하이브리드)
   - **시작일**: 첫날
   - **특수 사항** (선택): 원격 입사, 국제 신입, 경력/신입 구분
   - **기존 자료** (선택): 기존 온보딩 문서, 교육 자료

2. 프로젝트 루트에 `_workspace/` 디렉토리 생성
3. 입력값을 정리하여 `_workspace/00_input.md`에 저장
4. 기존 자료가 있으면 복사 후 해당 Phase 조정
5. 요청 범위에 따라 **실행 모드 결정**

### Phase 2: 팀 구성 및 실행

| 순서 | 작업 | 담당 | 의존성 | 산출물 |
|------|------|------|--------|--------|
| 1 | 온보딩 체크리스트 | 설계자 | 없음 | `_workspace/01_onboarding_checklist.md` |
| 2a | 교육 프로그램 | 개발자 | 작업 1 | `_workspace/02_training_program.md` |
| 2b | 멘토 가이드 | 매칭 전문가 | 작업 1 | `_workspace/03_mentor_guide.md` |
| 3 | 30-60-90 계획 | 추적자 | 작업 1, 2a, 2b | `_workspace/04_30_60_90_plan.md` |
| 4 | 경험 검토 | 검토자 | 작업 1, 2a, 2b, 3 | `_workspace/05_review_report.md` |

작업 2a(교육)와 2b(멘토)는 **병렬 실행**, 작업 1에만 의존합니다.

**팀 간 협업 흐름:**
- 설계자 완료 → 개발자에 학습 목표, 매칭 전문가에 역할 정의 전달
- 개발자 + 매칭 전문가 협력 → 멘토 교육 책임 영역 합의
- 개발자 + 매칭 전문가 완료 → 추적자에 교육 완료 기준, 멘토링 마일스톤 전달
- 추적자 완료 → 검토자에 30-60-90 목표, 피드백 프레임워크 전달
- 검토자가 모든 산출물 검증, 불일치/과부하 발견 시 수정 요청 (최대 2회)

### Phase 3: 통합 및 최종 산출

1. `_workspace/` 모든 파일 검증
2. 검토자의 일관성 검증 및 과부하 분석 결과 반영
3. 최종 요약 사용자에 보고

## 산출물

모든 산출물은 `_workspace/`에 저장됩니다:
- `00_input.md` — 사용자 입력 정리
- `01_onboarding_checklist.md` — 온보딩 체크리스트 및 일정
- `02_training_program.md` — 교육 프로그램
- `03_mentor_guide.md` — 멘토/버디 배정 가이드
- `04_30_60_90_plan.md` — 30-60-90일 계획
- `05_review_report.md` — 온보딩 경험 검토 보고서

## 오류 처리

| 오류 유형 | 대응 전략 |
|----------|----------|
| 직급 정보 부족 | 설계자가 직급별 표준 템플릿 제공 |
| 조직 규모 미상 | 소규모(약 20명), 중규모(약 100명), 대규모(100명+) 3가지 버전 제공 |
| 원격 신입 | 모든 항목에 원격 대체 활동 포함 |
| 에이전트 오류 | 1회 재시도 → 해당 산출물 없이 진행 |
| 과부하 발견 | 검토자가 활동 재배치 요청 → 재조정 (최대 2회) |
