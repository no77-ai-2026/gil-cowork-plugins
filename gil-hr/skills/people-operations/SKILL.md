## 원격 근무 정책 프레임워크

### 핵심 정책 영역

**1. 근무 형태 정의**
- 완전 원격(Full Remote): 사무실 출근 없음, 전국/해외 채용 가능
- 하이브리드: 주 N일 출근 의무, 나머지 재택 허용
- 원격 우선(Remote-First): 사무실 존재하되 원격이 기본 모드

**2. 코어 타임 설정**
- 코어 타임(필수 온라인): 예) 10:00-15:00 (팀 협업 집중)
- 플렉시블 타임: 코어 타임 외 자율 배치
- 타임존 정책: 해외 팀원 포함 시 겹치는 시간대 최소 3시간 확보

**3. 장비 및 환경 지원**
- 홈오피스 지원금 (예: 연 50-100만원, 모니터·의자·인터넷 등)
- 회사 지급 장비 기준 (노트북, 이어폰 등)
- 보안 요건 (VPN 의무, 화면 잠금 정책)

### 비동기 커뮤니케이션 가이드라인

**원칙**
- 기본 채널: 비동기(Slack 메시지, Notion 문서) 우선
- 즉시 응답 기대 금지: Slack 메시지는 2-4시간 내 응답이 기준
- 회의 대신 문서: 의사결정은 문서에 기록, 회의는 토론에만 사용

**효과적인 비동기 메시지 작성법**
```
[메시지 구조]
■ 맥락 (Background): 왜 이 메시지를 보내는가
■ 요청 (Ask): 구체적으로 무엇이 필요한가
■ 기한 (Deadline): 언제까지 필요한가 (없으면 명시)
■ 관련 자료 (Links): 참고할 문서나 링크

예시:
"[맥락] 다음 주 고객 미팅 준비 중입니다.
[요청] 지난달 제안서 최신본 공유 부탁드립니다.
[기한] 목요일 오전까지 받으면 됩니다.
[링크] 고객 미팅 노션 페이지: [URL]"
```

**회의 최소화 전략**
- 회의 전 체크: "이 회의가 문서로 대체 가능한가?"
- 회의 기본 시간: 25분 또는 50분 (패딩 타임 확보)
- 회의록: 회의 직후 5분 내 핵심 결정사항 Notion/Confluence 기록
- 정기 회의 감사: 분기 1회 모든 정기 회의 필요성 재검토

### 협업 도구 선정 기준

**팀 규모별 추천 스택**

| 팀 규모 | 커뮤니케이션 | 문서/지식 | 프로젝트 관리 | 화상회의 |
|---------|------------|---------|------------|---------|
| 1-10명 | Slack Free / 카카오워크 | Notion | Trello / Linear | Zoom / Google Meet |
| 11-50명 | Slack Pro | Notion / Confluence | Linear / Jira | Zoom |
| 51명 이상 | Slack Business+ | Confluence | Jira / Asana | Zoom Enterprise |

**도구 선정 체크리스트**
- [ ] 보안 인증 (SOC 2, ISO 27001) 보유 여부
- [ ] 데이터 저장 위치 (국내 서버 여부 — 금융·의료 필수)
- [ ] 기존 도구와의 연동 (API, Zapier 등)
- [ ] 모바일 앱 품질 (현장 직원 포함 시)
- [ ] 월 비용 예산 적합성

### 생산성 관리 프레임워크

**측정 지표 (Output 중심)**
- 작업 완료율 (주간 Task 완료율)
- 프로젝트 기한 준수율
- 품질 지표 (버그 수, 고객 만족도 등 직무별)
- 참여 지표 (회의 참여율, 문서 기여도)

**번아웃 예방 체크인 (주간)**
```
팀장이 팀원에게 매주 물어볼 3가지:
1. 이번 주 가장 에너지가 많이 들었던 일은 무엇이었나요?
2. 도움이 필요하거나 막히는 부분이 있나요?
3. 개인적으로 잘 되고 있는 것은 무엇인가요?
```

### 직원 경험 설계

**원격 팀 문화 활동**
- 가상 커피챗: 주 1회 랜덤 팀원 매칭 (15분 자유 대화)
- 비업무 채널: #daily-standup, #wins, #random 등 운영
- 분기 오프사이트: 원격 팀 전체 대면 모임 (1-2일)
- 온라인 팀 빌딩: 워드 클라우드, 온라인 게임, 퀴즈 등

**원격 온보딩 30일 계획**
```
1~3일: 장비 셋업, 계정 생성, 팀 소개 미팅
1주차: 도구 교육, 업무 프로세스 파악, 버디 배정
2주차: 첫 과제 수행, 일일 체크인 (버디와)
3~4주차: 독립 업무 시작, 주간 1:1 면담 시작
30일: 온보딩 회고 (기대 vs 현실, 개선점)
```

## 실행 규칙

1. 사용자 요청 수신 → 해당 영역 판별
2. `references/remote-work-ops.md` 존재 시 로드 → 전략 가이드에 따라 실행
3. `--deepthink` 또는 복잡 조직 설계 → `mcp__sequential-thinking__sequentialthinking` 호출
4. 결과물 생성 후 사용자 검토 요청

## 문제 해결

- **근로기준법 조항 불확실**: 재택근무 시 산재 적용, 근로시간 산정 등 법적 쟁점은 노무사에게 확인하세요. AI가 제공하는 내용은 참고용입니다.
- **직급 체계 정보 부족**: 팀 규모, 직무 유형, 기존 도구 현황을 알려주면 더 맞춤화된 정책을 설계할 수 있습니다.
- **보안 요건 불명확**: 금융, 의료, 공공 분야는 별도 보안 컴플라이언스(ISMS, HIPAA 등)가 필요합니다. 정보보안 담당자와 병행 검토하세요.
- **글로벌 팀 타임존 충돌**: 한국 기준 코어 타임 설정 시 타임존 매트릭스(겹치는 시간 시각화)를 요청하면 생성해 드립니다.

## 이 스킬을 사용하지 말아야 할 때

- **법적 노동 분쟁 대응**: 재택근무 중 발생한 산재, 임금 분쟁은 노무사에게 의뢰하세요.
- **채용·온보딩 프로세스 설계**: 채용 파이프라인과 신입 온보딩 계획은 `employment-manager` 스킬이 더 적합합니다.
- **성과평가 체계 설계**: OKR/KPI 설정 및 평가 면담은 `performance-review` 스킬을 사용하세요.
- **오퍼 레터·근로계약서 작성**: 계약 문서는 `draft-offer` 스킬을 사용하세요.
